# 优化版："扬公政策对比"模块实施计划（支持多版本管理）

根据您对“政策版本管理”的新需求，我调整了实施方案。核心思路是**不覆盖旧文件，而是每次修改都生成带时间戳的新文件**，从而建立完整的版本历史。

## 1. 核心文件存放位置 (变更)
建议在 `config` 下新建一个文件夹 **`config/policies/`** 来专门存放所有版本的政策文件。
- **初始操作**: 请将您整理好的 JSON 文件命名为 `policy_v1.json` (或任意名称) 放入 `config/policies/` 目录中。
- **自动版本化**: 每次编辑保存时，系统会自动在该目录下生成一个新文件，例如 `policy_20240118_103000.json`。

## 2. 后端开发 (server.cjs)
新增一组支持版本控制的 API：
- **GET /api/policy/versions**: 扫描 `config/policies/` 目录，按时间倒序返回所有可用的政策文件列表（供未来对比功能选择版本使用）。
- **GET /api/policy/latest**: 自动读取该目录下**最新**的一个 JSON 文件内容（作为“现行政策”的默认显示）。
- **POST /api/policy/save**: 接收编辑后的内容，**新建**一个带有时分秒时间戳的文件（如 `policy_20240118_142000.json`），确保历史版本不被覆盖。

## 3. 前端页面开发 (`src/pages/PolicyComparison/`)
- **页面 1: 现行政策 (`CurrentPolicy.jsx`)**
    - **加载逻辑**: 默认自动加载最新的政策文件版本。
    - **编辑功能**: 提供 JSON 编辑器。
    - **保存机制**: 点击“保存”时，提示“将生成新的政策版本”，确认后调用后端接口创建新文件。
    - **版本展示 (可选)**: 在页面上方显示当前正在编辑的文件名/版本号。
- **页面 2: 周报政策对比 (`WeeklyComparison.jsx`)**
    - **预留功能**: 界面占位。未来在此处添加“选择政策版本”的下拉框（数据来源即为 `/api/policy/versions`）。

## 4. 路由与菜单配置 (保持不变)
- **侧边栏**: 新增一级菜单“扬公政策对比”，下设“现行政策”和“周报政策对比”。
- **路由**: 对应 `/policy/current` 和 `/policy/comparison`。

这个方案完美满足了您的需求：既能让工作人员手动编辑，又能自动保留每一次修改的历史版本，为未来的“多版本对比”打下基础。待您确认，我即刻开始搭建。
