## 原因解释
- 这个报错来自 `JSON.parse(...)`：模型返回的内容**不是合法 JSON**，在某个字符串值里出现了“字符串没闭合”。
- 最常见触发方式：
  1) **字符串里出现了未转义的换行**（JSON 字符串里不能直接换行，必须写成 `\\n`）。
     - 你最近的提取提示词里我鼓励“内容用换行符拆多行”，模型很容易直接输出真实换行，导致 `Unterminated string`。
  2) 字符串里出现了未转义的引号（例如内容里含 `"` 没处理）。
  3) 输出被截断（超 token/超长度）导致 JSON 结尾缺失，也可能表现为类似错误。

## 为什么会发生在“某一批次”
- 分批时每批新闻的主题/短总结不一样：某批刚好包含更复杂的原文片段（引号、换行、表格），更容易让模型在 JSON 字符串里直接换行或插入未转义字符。

## 我将怎么修复（不让它再导致失败）
### 1) 提示词层面：明确禁止原始换行，改用转义序列
- 修改 `Policy Extraction Prompt`：
  - 明确要求：`内容` 字段如需分行，必须使用 `\\n`（两字符）表示换行，**不要**在 JSON 字符串里直接换行。
  - 同时强调：不得输出未转义的双引号。

### 2) 后端容错：在 JSON.parse 前做“最小修复”
- 在 `/api/policy/extract`：
  - 对模型返回文本做一次轻量清洗：将字符串内部的实际换行替换为 `\\n`（用状态机判断是否处于字符串中，避免误伤结构）。
  - 然后再 `JSON.parse`。

### 3) 兜底修复（可选但很稳）：失败时自动二次“修复JSON”
- 若仍 parse 失败：
  - 触发一次小模型调用：让模型把“上一轮输出”修复为合法 JSON（仍强制 `response_format: json_object`）。
  - 仅在失败时触发，避免平时额外成本。

## 验证
- 用你触发错误的那一批次复现：确认不会再出现 `Unterminated string...`，能继续分批跑完并合并。
- 同时确认提取结果中的 `内容` 分行表现为 `\\n`，前端显示时仍可读（必要时前端可把 `\\n` 渲染成换行）。
